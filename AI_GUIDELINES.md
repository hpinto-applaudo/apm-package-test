# AI-FILES-GUIDE.md

## 1. Mental Model

Think of the files as **layers of authority**:

```
Human Governance (non-negotiable)
├── AGENTS.md                          ← generated by apm compile
└── CLAUDE.md                          ← generated by apm compile --target claude
    ↓                                     (Claude Code session bootstrap; not in authority hierarchy)
Product & Engineering Constitution (rules of the system)
└── constitution.md                    ← generated by /speckit.constitution
    ↓
Execution & Assistance Behavior
└── copilot-instructions.md            ← initially deployed by SpecKit (Copilot integration);
                                          APM manages it once the marker line is present as the first line

The files above are what agents read.
The source of truth for AGENTS.md and CLAUDE.md is:
└── .apm/
    ├── instructions/                  ← always-on pointers, one per topic
    └── contexts/                      ← full content, loaded on demand
```

> In the APM workflow, `AGENTS.md` is a generated file produced by `apm compile` from source files under `.apm/`. Do not edit it directly: changes are overwritten on the next compile run. For repos migrating from a hand-authored `AGENTS.md`, see [guides/setup/migrate-to-apm.md](../guides/setup/migrate-to-apm.md).

Each file answers a **different question**:

| File                     | Core Question                                                       | How it is produced                                                                                                                |
| ------------------------ | ------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **AGENTS.md**            | _What is allowed in this repo and how humans + agents must behave?_ | Generated by `apm compile` from `.apm/` source files                                                                              |
| **CLAUDE.md**            | _What context does Claude Code load at session start?_              | Generated by `apm compile --target claude`; target-specific compiled output alongside `AGENTS.md`, not in the authority hierarchy |
| **Constitution**         | _How do we design, plan, and evolve software here?_                 | Generated by `/speckit.constitution` command                                                                                      |
| **Copilot Instructions** | _How should AI help me day-to-day while coding?_                    | Initially deployed by SpecKit (Copilot integration); APM manages it once the marker line is the first line of the file            |

## 2. AGENTS.md: Governance and Safety Contract

**Audience:** Humans + all AI agents  
**Scope:** Repo-wide, non-negotiable  
**Tone:** Strict, explicit, authoritative

> In the APM workflow, `AGENTS.md` is generated by `apm compile` from source files under `.apm/`. Do not edit it directly. The content requirements below apply to your `.apm/contexts/` files: they determine what ends up in the generated output. For legacy repos with a hand-authored `AGENTS.md`, see [guides/setup/migrate-to-apm.md](../guides/setup/migrate-to-apm.md).

### What AGENTS.md SHOULD contain

#### 1. Purpose of the repository

```
This repository contains production-grade software.
Changes must align with architectural, security, and quality standards.
```

#### 2. Allowed / forbidden actions for agents

```
Agents MAY:
- Read code
- Propose changes
- Create branches
- Generate tests

Agents MUST NOT:
- Push directly to main
- Modify CI/CD without approval
- Change security-sensitive files
- Introduce new dependencies without justification

```


#### 3. Branching & PR rules

```
- All changes via short-lived feature branches
- Every PR must link to a spec or task
- Draft PRs are allowed but must be closed or merged within X days

```

#### 4. Security & compliance rules

```
- No secrets in code
- Follow OWASP Top 10
- PII handling must comply with company policy
```

#### 5. Authority hierarchy

```
If instructions conflict:
1. AGENTS.md
2. Constitution
3. Copilot Instructions
```

#### What AGENTS.md should NOT contain

- General framework documentation: use MCP servers for live library docs instead
    
- Feature behavior or product requirements: those belong in specs
    
- Generic best practices that would apply to any codebase, not this one
    

**Note:** **Think of AGENTS.md as your “AI legal contract.”** Project-specific conventions, coding standards, architecture patterns, and quality gates all belong here: they inform how agents write code in your specific project. What does not belong is general framework documentation; use MCP servers for that instead.

## 3. GitHub Spec Kit Constitution: Engineering Law

**Audience:** Engineers + agents  
**Scope:** Product & system design  
**Tone:** Principled, stable, opinionated

This file is the **source of truth** for how software is designed and evolved.

---

### What the Constitution SHOULD contain

#### 1. Core engineering principles

```
- Simplicity over cleverness
- Explicit over implicit
- Evolutionary architecture
- Observability is mandatory

```

#### 2. Architecture rules

```
- Clean Architecture boundaries
- Domain logic is framework-agnostic
- No cross-module imports without interfaces
```

#### 3. Quality & delivery standards

```
- Every feature requires tests
- Public APIs must be versioned
- Backward compatibility is preferred

```

#### 4. Spec-driven development rules

```
- No code without a spec
- Specs must include:
  - Problem statement
  - Constraints
  - Non-goals
  - Acceptance criteria

```

#### 5. Branch & lifecycle expectations

```
- Features are developed in isolation
- Specs map 1:1 with feature branches
- Long-lived branches are prohibited

```

#### What the Constitution should NOT contain

- CLI commands
    
- Editor preferences
    
- “How Copilot should respond”
    
- Security permissions (that’s AGENTS.md)
    

**Note:** **This is your “Engineering Constitution.” It changes rarely.**

## 4. Copilot Instructions: Execution Behavior

**Audience:** Copilot / AI assistant  
**Scope:** Developer productivity  
**Tone:** Practical, tactical

This file is about **how AI behaves when helping you code**.

#### 1. Coding style & conventions

```
- Follow existing project structure
- Prefer small, composable functions
- Avoid premature optimization

```

#### 2. Language-specific guidance

```
Go:
- Use context.Context everywhere
- Return explicit errors
- No panics in application code

```

#### 3. Testing expectations

```
- Generate table-driven tests
- Prefer unit tests over integration tests
- Use mocks at boundaries

```


#### 4. Documentation behavior

```
- Update README if behavior changes
- Add comments only when intent is non-obvious

```

#### 5. What Copilot should ask before acting

```
- Clarify requirements if ambiguous
- Ask before adding new dependencies
- Ask before changing public APIs

```

#### What Copilot Instructions should NOT contain

- Governance rules
    
- Architecture laws
    
- Security policy
    
- Branching strategy
    

**Note:** **This file can evolve often. It’s safe to tweak.**


## 5. How the Files Relate

### Authority Flow

```
AGENTS.md
  └── defines what is allowed
Constitution
  └── defines how software is designed
Copilot Instructions
  └── defines how help is given

```

> `AGENTS.md` is what agents read. `.apm/` source files are what developers write.

### Practical Example

> Copilot wants to add a new dependency

- **AGENTS.md** → “Agents must not add dependencies without justification”
    
- **Constitution** → “Prefer standard library unless strong reason”
    
- **Copilot Instructions** → “Ask before adding dependencies”
    

Result: Copilot **asks first**, explains tradeoffs, follows architecture.



## 6. Final Rule of Thumb

| If the content is about…                                                                           | Put it in                                        |
| -------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| Safety, permissions, authority                                                                     | **`.apm/contexts/`** (compiles into `AGENTS.md`) |
| Architecture **principles** & SDLC law: design philosophy, non-negotiable engineering rules        | **Constitution**                                 |
| Architecture **patterns** & implementation conventions: coding standards, naming, day-to-day rules | **`.apm/contexts/`** (compiles into `AGENTS.md`) |
| How AI writes code                                                                                 | **`.apm/contexts/`** (compiles into `AGENTS.md`) |
| Agent setup, tooling, migration                                                                    | **`guides/setup/`**                              |
